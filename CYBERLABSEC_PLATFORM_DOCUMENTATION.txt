1. PLATFORM OVERVIEW & ARCHITECTURE
--------------------------------------------------------------------------------
CyberLabSec is a premium, dark-mode glassmorphic internal operations platform designed specifically for a world-class cybersecurity defense firm. It acts as the central nervous system for the company, managing everything from employee onboarding and task assignments to applicant tracking (ATS), secure internal communications, and performance analytics.

Tech Stack:
- Frontend: Next.js (App Router), React, Framer Motion (for micro-animations), Lucide React (icons), Canvas Confetti.
- Backend: Next.js API Routes (Serverless), Prisma ORM.
- Database: PostgreSQL (Neon DB).
- UI/UX: Custom CSS utilizing CSS Variables, flexbox/grid for extreme responsiveness, and glassmorphic aesthetic (blur, semi-transparent backgrounds).
- Auth: Custom JWT-based cookie authentication using `jose`. Roles are strictly binary: 'admin' (Company) and 'employee'.

3. COMPANY PORTAL (ADMIN / CEO DASHBOARD)
--------------------------------------------------------------------------------
Access: `/company/login` -> `/company/(authenticated)/*`
Role: Admins (e.g., Zohaib Khattak, CyberLabSec Admin)

This portal provides omniscient control over the entire organization.

MENUS & FLOWS:

1. Dashboard (`/company/dashboard`)
   - Purpose: High-level metrics and pulse of the company.
   - Flow: Fetches aggregated counts (Total Employees, Active Tasks, Pending Approvals) via Prisma.
   - Components: 
     - Top Stats Row (Cards showing counts).
     - Hiring Funnel (Conversion rates from Applied to Hired).
     - Live Activity Log (Real-time feed of employee actions, limited to 20 rows to prevent lag).

2. Job Postings (`/company/postings`)
   - Purpose: Manage open roles.
   - Flow: Admins can create, edit, pause, and close postings.
   - Buttons: "Create Posting" opens a modal -> "Save" immediately publishes to the public careers page.

3. Applications / ATS (`/company/applications`)
   - Purpose: Review candidates.
   - Flow: A Kanban board (or List view) showing candidates across stages (Reviewing, Invited, Selected, Hired, Rejected).
   - Interaction: Clicking a candidate card opens a detailed side/center modal.
   - Buttons:
     - "Shortlist & Invite" -> Updates status, generates an Interview Session token, and emails the candidate.
     - "Approve & Hire" -> Opens Offer Modal. Requires uploading a PDF offer letter, setting Start Date & Duration. Generates employee record instantly.
     - "Reject" -> Opens a confirmation window. If accepted, marks status as Rejected.
     - "Save Scorecard" -> Saves private admin notes and a 1-5 star internal rating.

4. Employees (`/company/employees`)
   - Purpose: Employee lifecycle management.
   - Flow: Shows all active and inactive employees.
   - Buttons:
     - "Direct Hire" -> Bypasses ATS. Creates an employee instantly, assigning them a unique `employeeCode` (e.g., CLS-001) and an initial temporary password.
     - "Terminate" -> Requires uploading a Termination PDF and adding a reason. Instantly revokes the employee's JWT token via the layout deactivation gate.
     - "Offboard" -> Graceful exit (resignation/end of contract). Generates a dynamic Experience/Recommendation Letter.

5. Workspace / Tasks (`/company/workspace`)
   - Purpose: Central task dispatch.
   - Flow: Admins create tasks assigned to Teams or Individuals. 
   - Rule: Task review is strictly binary. 
   - Buttons: 
     - "Approve" -> Closes the task, awards points to the employee, and updates leaderboard.
     - "Need more information" -> Pushes task back to the employee's To-Do list with admin feedback.

6. Teams (`/company/teams`)
   - Purpose: Organize employees into squads (e.g., Red Team, Blue Team, SOC).

7. Leave Requests (`/company/leave`)
   - Purpose: Approve/Reject employee time-off.
   - Flow: Shows a calendar or list of pending requests.
   - Logic: Prevents double-booking. Approval subtracts days from the employee's unwritten allowance (managed by HR logic).

8. Support Tickets (`/company/support`)
   - Purpose: Internal IT / HR helpdesk.
   - Flow: Admins read and respond to tickets created by employees.

9. Leaderboard (`/company/leaderboard`)
   - Purpose: Gamification. Ranks employees based on points earned from approved tasks and exceptional performance.

10. Announcements (`/company/announcements`)
    - Purpose: Broadcast company-wide or team-specific memos. Triggers push/email notifications.

5. CRITICAL LOGIC & SECURITY RULES (THE "RUTHLESS" STANDARD)
--------------------------------------------------------------------------------

A. Authentication Gatekeeping
- `lib/auth.ts`: Middleware extracts the JWT. `getAuthFromCookies()` then cross-references the database to ensure the employee's `status` is strictly `"Active"`. If "Terminated" or "Inactive", access is brutally blocked at the API and UI level.

B. The Deletion Cascade
- `prisma/schema.prisma`: Deleting a `JobPosting` cascade-deletes all associated `Applicant` and `OfferLetter` records to prevent orphan data and 500 Server Errors. Deleting a `Task` cascades to all `TaskSubmission` records.

C. Task Review Binary System
- To prevent operational ambiguity, tasks can only be "Approved" (Task closed, points awarded) or "Need more information" (Task returned). There are no messy middle states like "Partially Approved".

D. Mobile Responsiveness
- All data tables (Employees, Tickets) are wrapped in `overflow-x: auto` containers to prevent breaking the mobile viewport.
- Floating Action Buttons (FABs) on mobile (like "New Leave Request") have bottom padding to prevent overlapping with list items on iOS Safari.
